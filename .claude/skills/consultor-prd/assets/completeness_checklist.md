# Checklist de Completude do PRD

Use este checklist para verificar se o PRD está completo antes de entregar. Cada item deve estar presente e com conteúdo suficiente.

---

## Seções Estruturais

- [ ] Resumo Executivo — Contém visão geral compreensível por qualquer stakeholder
- [ ] Contexto e Background — Explica a situação atual e a motivação
- [ ] Declaração do Problema — Problema claro, com impacto e alternativas atuais
- [ ] Público-Alvo e Personas — Pelo menos 1 persona completa com contexto
- [ ] Proposta de Valor — Frase de valor + diferenciação + benefícios quantificados
- [ ] Objetivos e Métricas — Objetivos SMART com targets definidos
- [ ] Escopo do MVP — Inclui o que está dentro E o que está explicitamente fora
- [ ] Requisitos Funcionais — Cada funcionalidade tem descrição, persona e critérios de aceite
- [ ] Requisitos Não-Funcionais — Performance, segurança, escalabilidade documentados
- [ ] Regras de Negócio — Cada regra tem condição, ação e exceções
- [ ] Fluxos e Jornadas — Happy path detalhado + fluxos de exceção
- [ ] Integrações e Dependências — Sistemas externos listados com propósito e criticidade
- [ ] Restrições — Técnicas, operacionais, regulatórias e de negócio
- [ ] Riscos e Hipóteses — Riscos com probabilidade/impacto/mitigação; hipóteses com método de validação
- [ ] Priorização — MoSCoW ou RICE aplicado a todas as funcionalidades
- [ ] Dúvidas Pendentes — Lista com responsável e prazo
- [ ] Próximos Passos — Ações concretas com responsável e prazo
- [ ] Glossário — Termos técnicos ou de domínio explicados (se aplicável)
- [ ] Histórico de Revisões — Ao menos versão inicial registrada

---

## Qualidade do Conteúdo

- [ ] O problema está descrito do ponto de vista do USUÁRIO, não do sistema
- [ ] A proposta de valor é clara o suficiente para um não-técnico entender
- [ ] Os objetivos são mensuráveis (têm número, prazo e métrica)
- [ ] Os requisitos funcionais são verificáveis/testáveis
- [ ] As regras de negócio cobrem cenários de exceção
- [ ] Os fluxos cobrem caminhos de erro, não só o happy path
- [ ] Há distinção clara entre MVP e roadmap futuro
- [ ] Não há contradições entre seções
- [ ] Termos são usados de forma consistente ao longo de todo o documento

---

## Prontidão para SPEC

- [ ] Uma equipe técnica conseguiria iniciar SPECs com base apenas neste PRD
- [ ] Não há funcionalidades ambíguas que precisem de mais discovery
- [ ] As dependências técnicas estão identificadas e com status
- [ ] Os riscos técnicos têm plano de mitigação
- [ ] A sequência de implementação está sugerida
- [ ] Dúvidas pendentes têm prazo e não são bloqueantes para iniciar
